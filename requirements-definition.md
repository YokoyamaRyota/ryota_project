# 要求定義（ビジネス要求）

## 1. 背景
組織は、AIマルチエージェントのワークフロー（計画、実装、レビュー）を活用して開発スループットの向上を目指している。主要な事業リスクは以下である。
- プレミアムリクエストの増加
- 応答遅延の増加
- エージェント間の引き継ぎ時におけるユーザー意図からの逸脱

## 2. 目的
ユーザー要求を正確に実行しつつ、コストと応答時間を管理可能な範囲に維持する、VS Codeベースのエージェント開発システムを構築する。

## 3. ビジネス目標
- タスクのエンドツーエンドのリードタイムを短縮する
- レビューの一貫性と欠陥検出率を向上させる
- AI運用コストの予測可能性を高める
- 意図逸脱による手戻りを削減する

## 4. スコープ
スコープ内:
- VS Codeのカスタムエージェント/ワークフロー
- Coordinator、Planner、Implementer、Reviewerの役割分担
- 条件付き並列サブエージェント実行
- コストガードレールとフォールバックモード
- CI、課題管理、ドキュメント連携のための任意のMCP統合

スコープ外（Phase 1）:
- 全社レベルのガバナンス自動化
- VS Code以外のIDEを含む標準化
- 本番環境への自律デプロイ

## 5. ステークホルダー
- 開発者
- テックリード
- レビュアー
- セキュリティ/コンプライアンス担当
- 開発運用担当

## 6. 成功指標（SMART）
KPI-1 リードタイム:
- 基準値: 導入前の参考プロジェクトにおけるタスク完了時間の中央値（T0）
- 目標: 
  - Phase 1（0-4週）: T0 の 0.85 倍以下（システム安定化）
  - Phase 2（4-8週）: T0 の 0.75 倍以下（メモリ効果現れ始め）
  - Phase 3（8週以降）: T0 の 0.65 倍以下（蒸留パイプライン効果）
- 注記: 20タスク未満の初期フェーズではメモリ蒸留が未発動のため向上は緩い。段階的目標設定により実現可能性を向上
- 前提: Week 0 で baseline-metrics を取得し、Simple / Medium / Complex ごとの p50 / p90 を固定する
- 補足: Phase 1 終了時点で代表タスク数が 20 未満の場合、蒸留による改善評価は Phase 2 へ繰り越す
- 計測: タスク種別ごとの中央値と p90
- 計測除外:
	- ユーザー意思決定待ち時間
	- pending または suspended 状態の保留時間
- タスク種別定義:
	- 判定方針: 実運用上の複雑度判定は、変更行数・変更ファイル数に加えて、ユーザー要求の性質（簡易修正/大規模開発）、影響範囲、必要な検討観点を総合評価して確定する。
	- 計測バケット: KPI集計の一貫性のため、以下の閾値を計測バケットとして併用する。
	- Simple: 変更行数 < 80 かつ変更ファイル数 <= 2
	- Medium: 変更行数 80-300 または変更ファイル数 3-5
	- Complex: 変更行数 > 300 または変更ファイル数 >= 6

KPI-2 タスク単位コスト（Copilot Premium Requestベース）:
- Simpleタスク: premium_request_equivalent 1.0 以下
- Mediumタスク: premium_request_equivalent 2.0 以下
- Complexタスク: premium_request_equivalent 4.0 以下
- 定義: 1ユーザープロンプトあたりの消費量 = 1 x model_multiplier（Copilot公式倍率）
- 計測: 実行ログに基づくタスク単位の premium request 消費量集計。内訳：
	- execution_request_cost: 要求定義からレビュー完了までの実行工程
	- distillation_amortized_request_cost: 蒸留・イベント処理の均配（参考値）
	- event_audit_request_cost: イベント記録および再生検証
- 予算警告閾値: predicted_request_cost が remaining_request_budget の 80% 以上
- 注記:
	- Copilot のコスト算定はトークン長ではなく premium request と model_multiplier を用いる
	- Copilot Chat/Agent では1回のユーザープロンプトを1 request として計算する
	- トークン増加による追加コストは算定対象外とする
- 補足: Phase 1 では distillation_amortized_request_cost を参考値として記録し、受け入れ判定は execution_request_cost と event_audit_request_cost を優先する
- 実装制約: Copilot のプレミアムリクエスト使用量は VS Code からプログラム的に取得できないため、消費量計測は Coordinator によるタスクごとの推計値（planned_user_prompts × model_multiplier の累積）をベースとする。週次ガバナンスにて GitHub Copilot 使用量ページまたは GitHub API との照合を実施し、推計精度を検証する。

KPI-3 意図準拠率:
- 目標: 90% 以上
- 定義: goal、constraints、done criteria に重大な不一致がないこと
- 合格条件:
	- goal achieved = true
	- all must-have constraints satisfied = true
	- all must-have done criteria satisfied = true
	- no unapproved out-of-scope output = true
- 意図準拠率 = 合格タスク数 / 評価対象タスク総数
- 計測方法:
  - 自動判定: LLM（current generation model）で各タスク出力とcontract を比較
  - Spot-check: 毎週末、直近50タスクから無作為抽出5件を人間レビュアーが確認
  - 齟齬検出時: LLMスコアの信頼度を下げて再学習データとして記録

KPI-4 重大レビュー見逃し率:
- 目標: 仕込みテストシナリオに対して 5% 以下
- 定義: レビューパイプラインで検出されなかった既知の重大問題
- 重大問題の分類:
	- Security: 秘密情報漏えい、インジェクションリスク、認証回避
	- Correctness: データ破損、不正な状態遷移
	- Reliability: クラッシュ級の回帰、復旧不能なタイムアウトループ

KPI-5 意思決定応答性（システム制御部）:
- 目標: 複数案を提示してからユーザーへの質問発出までの中央値 3 分以内
- 注記: ユーザーの実際の回答時間（4時間SLA）はシステム制御外のため計測対象外
- 重要度: 低。本KPIよりも BR-06「調査投入時間＜8分」を優先制御する
- 計測: decision_pending 状態の実行ログ内での elapsed_time（M_0 → decision_gate_call の時間）
- メトリクス: weekly median, p95

KPI-6 悪化判定（現行方式比）:
- 定義: 現行の反復修正スタイルをベースラインとした時間/コストの悪化監視
- 警告閾値: 完了時間または premium_request_equivalent が現行方式比 +10% を超過
- 是正必須閾値: 完了時間または premium_request_equivalent が現行方式比 +20% を超過
- 評価単位: Simple / Medium / Complex
- 運用周期: 週次レビュー

## 7. ビジネス要求
BR-01 システムは全ワークフロー段階でユーザー意図を保持しなければならない。
BR-02 システムは速度だけでなくコストも同時に最適化しなければならない。
BR-03 システムは意思決定と再試行を追跡可能な監査ログとして残さなければならない。
BR-04 システムは予算超過、タイムアウト、ツール障害時に劣化運転で継続できなければならない。
BR-05 システムは自動収束に失敗した場合に人手へ引き継げなければならない。
BR-06 システムは新規必須機能の実装前に機能拡張の意思決定フローを実行しなければならない。
BR-07 システムは実現可能な複数の実装案とそのメリット/デメリットを提示し、後続の仕様化前にユーザーの明示的な方針決定を取得しなければならない。
BR-08 システムは、外部依存障害による連鎖遅延と無駄な再試行を抑えるため、外部連携呼び出しにサーキットブレーカ戦略を適用できなければならない。
BR-09 システムは、同一入力または同等差分の再実行時に、フェーズ結果を再利用して不要な推論を回避できなければならない。
BR-10 システムは、既知パターンの要求については意思決定ゲートを省略し、通常フローへ短絡できなければならない。
BR-16 システムは、意思決定完了後に、要求定義、要件定義、デリバリープラン、設計、実装、レビューの順で工程を進めなければならない。
BR-17 システムは、各工程で対応する成果物ファイルを作成または更新し、更新が完了するまで次工程へ進んではならない。
BR-18 システムは、下流工程で検出した問題の性質に応じて、要求定義、要件定義、デリバリープラン、設計のいずれかへ出戻りし、関連成果物を更新したうえで再開できなければならない。
BR-19 システムは、ユーザーから追加情報、意思決定、または明示的承認が本質的に必要な場合を除き、要求定義、要件定義、デリバリープラン、設計、実装、レビューの次工程へ自動で進行しなければならない。
BR-20 システムは、GitHub Copilotで利用可能なモデルの model_multiplier を用いて消費量を算出し、同一 multiplier の候補が複数ある場合は最も高性能なモデルを優先選択しなければならない。
BR-21 システムは、既存成果物（要求定義、要件定義、デリバリープラン、設計）が有効な場合、変更が必要な工程から開始できなければならない。
BR-22 システムは、外部知識参照時に参照元URLを記録し、公式ドキュメントまたは一次情報を優先し、外部送信データをマスキングしなければならない。
BR-23 システムは、記憶保持ポリシーとして基本TTLを30日とし、アーキテクチャ判断・重大バグ再発防止策・ユーザー明示方針・セキュリティ判断・外部依存採用理由を長期保持カテゴリとして管理しなければならない。

### 7.1 Phase別の必須範囲

Phase 1 で必須とする要求:
- BR-01 から BR-10
- BR-16 から BR-18
- BR-21
- BR-23
- BR-11 のうちエピソード記録
- BR-15 のうちイベント記録と再生可能性の最小構成

Phase 2 以降で段階的に有効化する要求:
- BR-12 蒸留パイプラインの自動実行
- BR-13 本格的なトークン予算最適化つき記憶取得
- BR-14 自動競合解決
- BR-22 外部知識参照ポリシーの本格運用

注記:
- Phase 1 では「記録できること」を優先し、「高度に最適化された再利用」は受け入れ条件に含めない。
- Phase 2 以降で Tier-2 / Tier-3 の活用、蒸留最適化、競合解決を段階有効化する。

## 8. リスクと対策
Risk-1 常時並列化によるコストの急増
- Mitigation: 複雑度と予算に応じた条件付き並列ポリシー

Risk-2 過剰なオーケストレーションによる遅延増加
- Mitigation: 小規模タスクは直列を既定とし、条件を満たしたタスクのみ並列化

Risk-3 エージェント間の意図逸脱
- Mitigation: 構造化ハンドオフ契約とフェーズ終了時の逸脱チェック

## 9. 継続的学習・メモリシステム要求

本書では、メモリシステムについて要求レベルのみを定義する。詳細なアーキテクチャ、パラメータ、イベント形式、形式保証、競合解決則は [memory-system-appendix.md](memory-system-appendix.md) に分離する。

### 9.1 背景と目的

システムは、タスク実行を通じて得られた知見を再利用可能な形で保持し、同種タスクの再実行時に品質向上と手戻り削減へつなげなければならない。

このとき、以下を同時に満たす必要がある。
- 現タスク情報を圧迫しないこと
- 記録内容の監査性と再現性を確保すること
- 誤った知見や古い知見を無制限に増やさないこと

### 9.2 要求原則

- 記憶は常時参照するコア知識と、必要時のみ参照する補助知識に分離する。
- タスク完了時には、少なくとも成果、判断根拠、失敗パターンを構造化記録する。
- メモリ変更はイベントとして記録し、過去状態を追跡可能にする。
- 蒸留、競合解決、検索最適化は段階的に有効化し、Phase 1 の受け入れ条件には含めない。
- 記憶保持は基本TTL（30日）と長期保持カテゴリを分離し、誤記憶の訂正・削除要求を受け付ける。

### 9.3 追加ビジネス要求

BR-11 システムはタスク完了後に、成果・判断根拠・発見した失敗パターンをエピソード記憶として構造化記録しなければならない。

BR-12 システムはエピソード記憶が一定数に達した場合、蒸留パイプラインを実行してセマンティック記憶へ統合し、エピソードファイルをアーカイブしなければならない。

BR-13 システムは記憶取得時にコンテキストトークン予算を管理し、現タスク情報が記憶によって圧迫されないよう取得件数を動的に制限しなければならない。

BR-14 システムは同一パターンへの矛盾する記憶が存在する場合、タイムスタンプおよび特異性に基づいて競合を解決し、解決不能な場合はユーザーへ提示しなければならない。

BR-15 システムは、メモリ変更イベントをすべてタイムスタンプ付きで記録し、初期状態とイベントシーケンスから任意の時刻の完全なメモリ状態を再構成できなければならない。

### 9.4 メモリシステムの主要リスク

- 蒸留品質の低下により、有用知識が失われるリスク
- 埋め込みモデル変更により、既存索引の整合性が崩れるリスク
- 誤ったタスク結果が学習されるメモリポイズニングのリスク
- イベントログ肥大化により、再生コストが増大するリスク
- 形式保証や再生検証が過剰コストになるリスク

各リスクの詳細な対策、運用パラメータ、イベント仕様、形式保証は [memory-system-appendix.md](memory-system-appendix.md) を参照する。

## 10. テスト戦略

### 10.1 KPI-4「重大レビュー見逃し率5%」の達成方法

**仕込みテストシナリオ**

レビュー見逃しを計測するため、以下の3カテゴリで既知脆弱性を意図的に埋め込む：

| カテゴリ | 脆弱性パターン | 埋込数 | 検出方法 |
|---------|--------------|------|--------|
| Security | `password = input()` を含むコード、hardcoded credentials | 5件/月 | マスク語句検索 |
| Correctness | 不正な状態遷移（例：初期化前アクセス）、type mismatch | 5件/月 | 静的解析 |
| Reliability | 無限ループ、例外未処理、リソースリーク | 5件/月 | タイムアウト検知 |

**計測方法**

- レビュー実行後、上記15件の脆弱性うち、検出されたもの数をカウント
- 月次見逃し率 = (15 - detected) / 15
- 目標: 見逃し率 < 5% ⇔ detected > 14件（14/15）

### 10.2 回帰テスト自動化

**対象フェーズ**

- Fast Gate（必須チェック）: 100%カバレッジ → 既知パターン脆弱性テスト + git 差分テスト
- Deep Review（任意チェック）: サンプリング50% → 週1回、直近実運用タスクから無作為10件

**テストケース生成**

- Drift Detector エラーをテストケース化:
  ```
  if hard_drift detected:
    create_regression_test(task_contract, output, drift_details)
  ```
- アーカイブ: memory/test_cases/ 下に保存、月1回整理

## 11. 受け入れ条件（ビジネス観点）
- 代表的なタスクを、コスト目標内で段階的目標より高速に完了できること（KPI-1 Phase 1達成）。
- パイロット期間（4週）で意図準拠率目標の75%以上を達成していること（KPI-3 > 67.5%）。
- 監査ログにより失敗や手戻りの原因分析が可能であること（BR-03, BR-15 実装済み）。
- フォールバックと手動引き継ぎの運用手順書が整備されていること（セクション 12）。
- 仕込みテストシナリオ実施により KPI-4 の計測方法が確立されていること。
- 悪化判定が週次で実施され、+20%超過時の是正アクションが運用されていること（KPI-6）。

## 12. 運用ガイダンス

### 12.1 suspended 状態での記憶処理

人手引き継ぎ（suspended）中のメモリアクセスポリシー：
- **許可**: memory/core.md, memory/patterns/ の読み取り（ユーザーがレビューして参照）
- **禁止**: episodes/ への新規書き込み（手動作業がメモリを「学習」することは回避）
- **推奨**: suspended 解決後、タスク全体をコーディネータで re-executeするか人手修正かを選択
  - re-execute の場合: 新規 task_id で実行（旧エピソードは archive のまま）
  - 人手修正の場合: 修正内容を incident 記録し、重要なら蒸留に反映

### 12.2 ロールバック時のメモリ処理

機能を巻き戻す場合（新機能 X の削除）：
1. イベント層でロールバック前の timestamp T_X を特定
2. memory の状態を T_X 時点の状態へ復元（イベントリプレイ）
3. T_X 以降の蒸留済みエントリで「機能X関連」を含むものはauto-tag
4. 新モデル導入時に、タグ付きエントリを優先的に再蒸留候補とする

### 12.3 Phase 1（0-4週）実装ロードマップ

| Week | マイルストーン | 実装項目 |
|------|-------------|----------|
| 0 | ベースライン固定 | 代表タスクを選定し baseline-metrics を取得。Simple / Medium / Complex の p50 / p90 を確定 |
| 1 | システム起動 | Coordinator, Planner, Implementer のみ（メモリシステム無効） |
| 2 | イベント層有効化 | audit_log/ 記録開始。リプレイテスト開始（サンプリング） |
| 3 | 記憶書き込み有効化 | タスク完了時に episodes/ 記録。蒸留はまだ手動トリガー |
| 4 | 蒸留パイプライン限定検証 | feature flag 配下で蒸留を限定有効化（手動トリガー含む）。失敗時の再実行性を確認 |

Phase 1 の週次完了条件:
- Week 1 完了: known_pattern タスク3件以上を通し、phase transition と drift 記録が監査ログで確認できる
- Week 2 完了: イベント再生サンプリングが成功し、欠損イベント 0 件を確認できる
- Week 3 完了: hard drift 未解消タスクを episodes/ に記録しないことを確認できる
- Week 4 完了: distillation を限定有効化し、失敗時に episodes/ を保持したまま再実行できる（本番の自動実行は Phase 2 以降）

### 12.4 LLMモデル変更時の処理

新モデル導入時のメモリ再検証：
1. 新モデルで既存 Tier-2 パターンを re-evaluate
2. 信頼度低下したエントリ（> 0.1 差分）を identify
3. 低信頼度エントリを archive に移動 or 再蒸留キューに追加
4. 新モデルで新規エピソード蒸留開始

### 12.5 開発ライフサイクルと出戻り方針

本システムの標準的な開発順序は以下とする。

1. 要求の明確化と要求分類
2. 必要に応じた調査、複数案提示、意思決定
3. 要求定義の作成または更新
4. 要件定義の作成または更新
5. デリバリープランの作成または更新
6. 設計の作成または更新
7. 実装
8. レビュー
9. 完了判定と記録

開始工程の選択ルール:
- 既存成果物が有効で、変更範囲が限定される場合は変更が必要な工程から開始してよい。
- レビュー依頼のみの場合はレビュー工程から開始してよい。
- 設計変更が要件定義の範囲内で閉じる場合は設計工程から開始してよい。
- 要求定義または要件定義に影響する変更は、該当する上流工程へ戻って更新しなければならない。

各工程で更新する主要成果物は以下とする。

| 工程 | 必須成果物 | 目的 |
|------|------------|------|
| 要求定義 | requirements-definition.md | ビジネス目的、制約、成功条件、スコープを確定する |
| 要件定義 | system-specification.md | 機能要件、非機能要件、受け入れ基準を確定する |
| デリバリープラン | delivery-plan.md | 実装順序、マイルストーン、依存関係、検証方法を定義する |
| 設計 | design.md | 制御方式、主要コンポーネント、実装方針を整理する |
| 実装 | 対象コード・設定・補助成果物 | 設計に基づき変更を反映する |
| レビュー | review-report.md、実行ログ、メモリ記録 | 品質判定、出戻り要否、次アクションを確定する |

出戻りルール:
- 要求の目的、制約、スコープが不明確または矛盾した場合は、要求定義へ戻る。
- 受け入れ基準不足、must-have 制約の欠落、実現不能要件の発覚時は、要件定義へ戻る。
- 実装順序、依存関係、マイルストーンの見積誤りが判明した場合は、デリバリープランへ戻る。
- アーキテクチャ不整合、非機能要件未達、レビューで設計起因の欠陥が見つかった場合は、設計へ戻る。
- 実装不備のみで解消可能な場合に限り、実装工程へ戻って修正する。

進行制約:
- 下流工程へ進む前に、直前工程の成果物更新を完了していなければならない。
- 出戻りが発生した場合、影響を受ける下流成果物は再確認し、必要に応じて更新する。
- レビュー完了後に要求・要件・設計へ影響する問題が見つかった場合、実装だけで閉じてはならない。
- ユーザーから追加情報、意思決定、または実行権限の確認が本質的に必要な場合を除き、次工程への進行可否を都度確認してはならない。
- たとえば、通常の成果物更新、次工程への遷移、既定ポリシーに基づくレビュー実行については、個別の許可を求めず自動で進行する。

## 13. 機能拡張の意思決定フロー
新しい必須機能が要求された場合、次の手順を必須とする。
1. 機能の意図、制約、成功条件を明確化する。
2. 実装不確実性がある場合は、対象を絞ったインターネット調査を行う。
3. 実現可能な実装方式を最低2案作成する。
4. 各案について、メリット、デメリット、リスクレベル、予想遅延影響、予想コスト影響を提示する。
5. ユーザーに優先方針の選択（またはハイブリッド方針の指定）を求める。
6. ユーザー方針確定後に、以下を生成/更新する。
	- 要求定義
	- 要件定義
	- デリバリープラン
	- 設計
7. 上流成果物が確定した後にのみ、実装とレビューへ進む。

意思決定フローの品質ゲート:
- ユーザー方針が確定するまで、実装計画や設計の最終確定を行ってはならない。
- 上流成果物が未更新の状態で実装に進んではならない。
- ユーザーの追加情報が不要であり、既存方針で判断可能な場合は、要求定義、要件定義、デリバリープラン、設計、実装、レビューの各工程を自動継続しなければならない。

調査実施の判定ルール:
- 以下のいずれかを満たす場合は調査を実施する。
	- 現在のリポジトリ内に類似実装がない
	- ライブラリ/フレームワークの実現性が不明
	- ローカル知識だけでは統合リスクを見積もれない
- 以下をすべて満たす場合は調査を省略できる。
	- 実証済みの内部実装パターンがある
	- 未解決の技術的不確実性がない

調査予算とソースポリシー:
- 時間予算: 1機能要求あたり最大8分
- クエリ予算: 最大6件の焦点化クエリ
- 優先ソース:
	- ベンダー/フレームワークの公式ドキュメント
	- 標準/仕様書
	- 成熟したオープンソースリポジトリ
- ソース品質の最低条件:
	- バージョンや日付が明示されていること
	- 実装に落とし込める具体性があること
	- 再現可能な手順が示されていること

調査ソース件数の適用ルール:
- 高リスク変更（認証、秘密情報、課金、データ整合性）: 独立ソース2件以上を必須
- 低〜中リスク変更: 独立ソース1件以上で可

調査結果の記録ルール:
- 調査で採用した情報源はURLを記録する。
- 参照先は公式ドキュメントまたは一次情報を優先し、二次情報を採用する場合は補助的根拠として扱う。
- 外部参照時に送信するデータは、機密・識別子・秘密情報をマスキングした要約に限定する。

## 14. 改訂採用ポリシー（設計有効性重視）
本要件は、将来設計で判断しやすいように、機能追加を次の基準で採用する。

採用条件（すべて満たす）:
- 期待効果が速度またはコストで定量的に確認可能
- 既存KPI（KPI-1〜KPI-4）の少なくとも1つを改善する見込みがある
- 実装コストに対して運用効果が継続的に見込める

見送り条件（いずれかを満たす）:
- 効果が主観的で、KPIや受け入れ基準で検証できない
- 処理遅延やプレミアム消費の増加が改善効果を上回る
- Preview機能への依存が強く、代替手段がある

本改訂で優先採用する機能:
- 外部連携呼び出しのサーキットブレーカ
- フェーズ結果キャッシュ
- 並列実行上限の動的制御

本改訂で必須を緩和する項目:
- すべての機能調査で独立ソース2件必須という運用は採用しない
  - 理由: 低リスク案件での速度低下が大きく、コスト効率を悪化させるため

意思決定記録の最小項目:
- feature_id
- considered_options
- selected_option
- user_rationale
- expected_latency_impact
- expected_cost_impact

ガバナンス責任:
- 週次KPIレビュー責任者: engineering lead
- 閾値変更の承認者: tech lead または指定レビュアー
- ロールバック条件: 閾値変更後にKPIが 10%以上悪化した場合、1レビューサイクル以内に直前閾値へ戻す

## 15. GitHub Copilot 専用運用前提（最新版）

本書は VS Code 上の GitHub Copilot を前提とする。以下は設計・実装・運用時の拘束条件とする。

### 15.1 リクエスト課金と算定ルール

- Copilot Chat（ask/edit/agent/plan）では、1回のユーザープロンプトを 1 request として扱う。
- premium request 消費量は、`request_count x model_multiplier` で算定する。
- Agent 実行中の内部ツール呼び出しは、ユーザープロンプト追加がない限り premium request を加算しない。
- トークン量増加は premium request 算定に含めない。

### 15.2 Copilotプランとモデル前提

- 利用可能モデルは Copilot プランおよびクライアント（VS Code）に依存する。
- paid plan では included model（例: GPT-5 mini, GPT-4.1, GPT-4o）を優先して利用し、要件を満たせない場合のみ premium model を使用する。
- premium request が枯渇した場合は included model へフォールバックして継続する。
- premium request カウンタは月次リセットされ、未使用分は繰り越さない。

### 15.3 モデル倍率の扱い

- model_multiplier は GitHub公式の supported models / requests ドキュメントを単一の正として参照する。
- 倍率や提供モデルは変更され得るため、仕様書に固定値を埋め込まず、運用時に最新値を同期する。
- 同一 multiplier の候補が複数ある場合は、まず最大性能モデルを選択し、同等性能の場合のみ Copilot model comparison の推奨タスク適合性で決定する。

### 15.4 仕様有効性の境界

- 本書の timeouts、SLA、降格順序は「ローカルオーケストレーションの制御目標」であり、GitHub Copilot サービス自体のSLAを保証するものではない。
- Copilot側の提供仕様変更（モデル退役、倍率変更、機能提供範囲変更）が発生した場合、本書は次回週次ガバナンスで改訂する。

## 16. 工程ゲート定義（Entry / Exit）

本節は、各工程への着手条件（Entry）と完了条件（Exit）を定義し、Go / No-Go 判定を明確化する。

| 工程 | Entry 条件 | Exit 条件 | 判定責任 |
|------|------------|-----------|----------|
| 要求定義 | ユーザー要求の背景、目的、制約、優先度が入力済み | BR と KPI が矛盾なく定義される | coordinator / user |
| 要件定義 | 要求定義が確定し、優先順位が合意済み | 機能要件、非機能要件、受け入れ基準が確定 | planner / tech lead |
| デリバリープラン | 要件定義が確定済み | マイルストーン、依存関係、検証計画が確定 | planner |
| 設計 | 要件と計画が確定済み | 設計方針、境界、失敗時方針、計測ポイントが確定 | architect role |
| 実装 | 設計が確定済み | 実装差分、テスト結果、レビュー準備が完了 | implementer |
| レビュー | 実装とテスト証跡が提出済み | 重大欠陥対応、要求適合確認、次アクション確定 | reviewer |

Go / No-Go 基準:
- Go: Exit 条件がすべて満たされ、重大未解決事項が 0。
- No-Go: Exit 条件未達、または重大欠陥が未解決。

## 17. テスト責務分解（標準フロー）

既存のレビュー中心戦略に加えて、テスト責務を次の層で定義する。

| 層 | 目的 | 最低実施基準 | ブロック条件 |
|----|------|--------------|--------------|
| Unit | ロジック単位の正当性 | 変更箇所に対するテスト追加または更新 | 失敗テストが残存 |
| Integration | コンポーネント間接続の整合 | 主要連携パスの成功確認 | 主要連携の失敗 |
| E2E/Scenario | ユーザー要求の達成確認 | 複雑度別の代表シナリオを実行 | must-have シナリオ失敗 |
| Security | 重大脆弱性の予防 | 既知パターン検査 + リスク変更の重点確認 | 重大項目検出で未対処 |
| Performance | 応答性と遅延抑制 | 主要フローの応答計測 | p95 目標未達（例外承認なし） |

注記:
- Phase 1 は Unit / Integration / 代表シナリオを必須とし、E2E 全面適用は段階有効化する。
- Security と Performance は変更リスクに応じて重点度を切り替える。

## 18. トレーサビリティ管理

UR から BR、KPI、テスト、レビュー証跡までの追跡可能性を確保する。

管理ルール:
- 各 BR は少なくとも 1 つ以上の UR に紐づくこと。
- 各 KPI は、対応する BR と評価シナリオを持つこと。
- レビュー報告には、対象 BR と結果（pass/fail/conditional pass）を記録すること。

最小トレーサビリティ項目:
- source_ur_id
- mapped_br_id
- mapped_kpi_id
- validation_test_id
- review_evidence_id

## 19. 非機能要件（NFR）しきい値

通常ソフトウェア開発で不足しやすい NFR を運用可能なしきい値で定義する。

| 区分 | 指標 | 目標値（初期） | 測定タイミング |
|------|------|----------------|----------------|
| 応答性 | decision_gate 発出までの時間 | median 3分以内、p95 8分以内 | 週次 |
| 安定性 | 途中失敗からの再開成功率 | 95%以上 | 週次 |
| 可観測性 | 監査ログ欠損率 | 1%未満 | 週次 |
| 継続性 | 外部障害時の縮退継続率 | 90%以上 | 月次 |
| 品質維持 | 重大レビュー見逃し率 | 5%以下 | 月次 |

注記:
- 目標値は初期運用値であり、週次ガバナンスで見直し可能。

## 20. 変更管理（Change Request）

要求・要件・設計の変更は Change Request として管理する。

手順:
1. 変更要求を起票し、変更理由と期待効果を記録。
2. 影響分析（BR、KPI、成果物、テスト）を実施。
3. 承認者（tech lead または reviewer）が採否を決定。
4. 承認時は対象成果物を更新し、トレーサビリティを再同期。
5. 却下時は理由を記録し、代替案を提示。

版管理ルール:
- 要求定義の意味変更を含む場合は minor 以上を更新。
- 閾値変更や判定規則変更は changelog に根拠を残す。

## 21. 運用 Runbook（最小運用）

日常運用の監視と一次対応を定義する。

日次確認:
- 失敗タスク件数
- 手動引き継ぎ件数
- 予算警告発生件数

週次確認:
- KPI-1 から KPI-6 の達成状況
- 悪化判定（+10% 警告、+20% 是正）の発生状況
- 記憶品質（誤記憶訂正件数、長期保持カテゴリ更新）

エスカレーション:
- 是正必須閾値超過が 2 週連続した場合、運用方針を臨時レビューする。
- 重大欠陥見逃しが発生した場合、即時に要件または設計工程へ出戻りする。

## 22. UAT（ユーザー受け入れ）シナリオ

複雑度別に最低 1 シナリオ以上を維持し、代表タスクで受け入れ判定を行う。

| 区分 | 代表シナリオ | 合格条件 |
|------|--------------|----------|
| Simple | 既存機能の小修正 | 要求どおり修正され、再修正ラウンド 1 回以内 |
| Medium | 要件範囲内の設計変更 | 設計変更が要件逸脱なく実装反映される |
| Complex | 複数要素を含む機能追加 | 要求適合、重大欠陥なし、KPI悪化閾値内 |

UAT 判定:
- ユーザー最終判定を正とし、判定理由を review-report に記録する。
- 不合格時は、原因を要求/要件/設計/実装のいずれかに分類して再実行計画を作成する。
